# 学生端考试大纲页面改版方案（2026-03-23）

## 1. 阶段与结论

- 三阶段路由结果：`开发前（readiness）`
- 推荐技能：`software-development-readiness-governance`
- 当前结论：允许继续细化方案，但进入实现前必须冻结页面结构、交互状态、验收口径与实施边界。

本次任务不是直接修改代码，而是把学生端“我的题库 > 考试大纲”页面从“能展示”调整为“更容易找、看、比、记”的可实施方案，作为后续开发输入基线。

## 2. 页面定位冻结

### 2.1 保留定位

- 页面继续保留在：`我的题库 > 考试大纲`
- 页面继续保持：`全科总览 + 单科查看`
- 页面继续保持只读，不承接任何写操作
- 页面继续不跟随当前练习科目自动切换，避免把“总览页”做成“上下文页”

### 2.2 调整方向

- 从“展示全部科目和脑图”升级为“先快速定位科目，再快速定位知识结构”
- 从“全展开阅读”升级为“搜索、定位、展开、对照”
- 从“目录和详情松散串联”升级为“目录与详情强联动”

## 3. 当前页面主要问题

### 3.1 首屏不是操作页

当前页面首屏先展示页头和 4 张概览卡，再进入筛选和科目目录。用户的首要动作其实是“找目标科目”，但页面把概览信息放在了操作之前。

结果：

- 首屏有效操作密度低
- 搜索与筛选入口不够靠前
- 页面更像汇报页，不像查阅工具

### 3.2 文案与布局不一致

当前页面文案强调“右侧查看该科目的完整大纲脑图”，但实现结构实际是“上方选科目，下方看详情”。桌面端切换科目后，用户仍需继续向下滚动才能看到新内容。

结果：

- 操作反馈不够即时
- 科目切换与详情变化不在同一视觉区域
- 用户会误以为未切换成功

### 3.3 搜索范围过窄

当前搜索仅支持：

- 科目名称
- 科目编码
- 考试门类
- 联考专业组

不支持搜索大纲正文中的章节或知识点。

结果：

- 学生知道“要找哪个知识点”时，仍然要先猜属于哪门课、再展开阅读
- 页面缺少知识点定位能力

### 3.4 当前选中科目容易丢失

筛选条件变化后，如果当前选中科目不在结果集里，页面会自动切换到第一个科目。

结果：

- 用户上下文被打断
- 页面内容会突然跳变
- 容易造成“我刚刚看的那门课去哪了”的认知负担

### 3.5 缺少页面级错误态

当前加载失败只通过消息提示反馈，没有页面级错误容器和重试动作；用户可能看到空页面或无内容区域，却无法判断是“无数据”还是“加载失败”。

结果：

- 空态和错误态语义混淆
- 页面可靠性表达不足

### 3.6 脑图可读性不足

当前“具体内容与要求”采用递归卡片全展开方式展示，适合小科目，不适合节点多、层级深的长大纲。

结果：

- 首次进入时信息量过大
- 缺少折叠、目录锚点、回到当前位置等能力
- 移动端阅读成本更高

### 3.7 缺少版本信任信息

接口与数据中已有：

- `generatedAt`
- `policyVersionCode`
- `sourceFile`

但页面未向学生展示这些信息。

结果：

- 用户无法确认看到的是哪一版大纲
- 页面缺少“内容可信”证据

## 4. 改版目标

本次改版只解决以下 4 件事：

1. 让学生进入页面后先能快速找到目标科目。
2. 让学生能在单科内快速定位章节和知识点。
3. 让目录、筛选、详情、脑图之间形成联动闭环。
4. 让页面具备完整的加载、空态、错误态和移动端阅读方案。

本次明确不做：

- 不把考试大纲页改成练习入口页
- 不在本页新增收藏、标注、笔记、勾选完成等写操作
- 不新增大纲编辑能力
- 不引入新的后端写接口

## 5. 改版后的信息架构

### 5.1 桌面端结构

页面改为三段式：

1. 顶部轻量页头
2. 中部双栏工作区
3. 底部补充说明区

双栏工作区结构如下：

- 左栏：科目目录区
- 右栏：当前科目详情区

左栏负责“找”，右栏负责“看”。

### 5.2 顶部轻量页头

保留页面标题 `考试大纲`，但弱化现有大卡片概览，改成单行信息摘要：

- 总科目数
- 总节点数
- 当前政策版本
- 最近生成时间

目标：

- 保留总览感
- 不抢占首屏操作空间

### 5.3 左栏目录区

左栏包含：

- 关键字搜索
- 科目类型筛选
- 考试门类筛选
- 可滚动科目列表

目录卡片只保留最关键 4 项信息：

- 科目名
- 公共课 / 专业课
- 考试门类
- 联考专业组或“全局共享”

选中项要有强对比态，并在切换后保持可见。

### 5.4 右栏详情区

详情区分为四块：

1. 科目头部信息
2. 快速导航区
3. 考试形式与题型摘要
4. 具体内容与要求

科目头部信息展示：

- 科目名
- 科目类型
- 考试门类
- 联考专业组
- 节点数量
- 大纲版本
- 数据生成时间

### 5.5 快速导航区

新增“章节导航”或“目录锚点”区，用于快速跳转到：

- 科目简介
- 考试形式与参考题型
- 具体内容与要求下的各 L3 主章节

目标：

- 避免长页纯滚动查找
- 帮助学生在长大纲中快速定位

## 6. 关键交互方案

### 6.1 科目切换

- 点击左栏科目卡片后，右栏详情即时切换
- 桌面端不需要继续滚动到下方才能看到结果
- 切换后右栏滚动位置重置到详情顶部

### 6.2 搜索分两层

新增两种搜索语义：

- 科目搜索：用于找某一门课
- 大纲内容搜索：用于找某个章节或知识点

建议交互：

- 默认先显示科目搜索
- 进入单科详情后，在右栏增加“在当前科目大纲中搜索”输入框
- 搜索命中后自动高亮并展开对应分支

### 6.3 脑图从“全展开”改为“渐进展开”

默认规则：

- L3 默认展开
- L4 默认折叠
- L5 按需展开

补充动作：

- 展开全部
- 收起全部
- 回到顶部
- 定位到当前命中章节

### 6.4 当前选中上下文保留

筛选变化时遵守以下规则：

- 如果当前科目仍在结果中，保持不变
- 如果当前科目不在结果中，显示“当前科目已被筛选隐藏”提示，并允许一键清除筛选
- 不再静默跳到第一项

### 6.5 详情与导航联动

- 点击章节导航项，右栏滚动到对应章节
- 手动滚动正文时，当前导航项同步高亮
- 大纲内容搜索命中后，导航区同步定位到命中章节

## 7. 页面状态矩阵

### 7.1 加载态

- 首次进入页面时，左栏目录与右栏详情均显示 skeleton 或 loading 容器
- 不使用只遮罩整页、不体现结构的模糊 loading

### 7.2 空态

分 2 类空态：

1. 目录筛选为空
   - 文案：`当前筛选条件下没有匹配的考试科目`
   - 动作：`重置筛选`
2. 当前科目缺少大纲结构
   - 文案：`当前科目暂未识别到“具体内容与要求”结构`
   - 动作：`查看其他章节摘要`

### 7.3 错误态

接口加载失败时展示页面级错误卡：

- 文案：`考试大纲加载失败，请稍后重试`
- 动作：`重新加载`

页面错误态不得与空态共用同一容器语义。

### 7.4 无搜索命中态

在当前科目内搜索正文无命中时：

- 保持原大纲内容不消失
- 在搜索框附近提示：`未找到匹配知识点，请尝试更换关键词`

### 7.5 移动端态

移动端结构改为：

- 顶部页头
- 科目选择抽屉或折叠目录
- 当前科目详情单列流

移动端规则：

- 默认优先展示当前科目详情
- 科目目录收进抽屉或折叠面板
- 脑图默认更克制，只展开 L3 主章节

## 8. 文案与认知口径

本页统一使用以下语义：

- 页面标题：`考试大纲`
- 左栏标题：`全部考试科目`
- 右栏标题：`当前科目大纲`
- 搜索占位文案：
  - 左栏：`搜索科目名称 / 科目编码 / 门类 / 专业组`
  - 右栏：`在当前科目中搜索章节或知识点`

避免继续使用：

- `脑图总览`
- `全景图谱`
- `探索`

本页定位是“查阅和定位”，不是“探索和沉浸浏览”。

## 9. 实施边界

### 9.1 接口边界

优先复用现有只读接口：

- `GET /api/question-bank/student/syllabus/catalog`

如需支持“当前科目大纲内容搜索”，优先前端基于已有 `nodes` 做本地检索；只有在性能不可接受时，才评估新增只读搜索接口。

### 9.2 组件边界

优先在以下文件内改造：

- `frontend/src/views/Student/QuestionBankSyllabus.vue`
- `frontend/src/components/student/QuestionBankSyllabusBranch.vue`
- `frontend/src/utils/studentSyllabusAtlas.js`

如有必要，可新增：

- 科目目录组件
- 章节导航组件
- 大纲搜索命中高亮组件

### 9.3 风格边界

- 继续沿用学生端蓝灰基调与现有 token
- 不新开平行视觉主题
- 不做高饱和花哨脑图
- 不增加大面积统计看板

## 10. 验收标准

### 10.1 首屏可操作性

- Given 用户进入“我的题库 > 考试大纲”
- When 页面完成加载
- Then 用户首屏应先看到搜索、筛选和科目目录，而不是多张大卡片抢占主流程

### 10.2 科目切换反馈

- Given 用户位于桌面端考试大纲页
- When 用户点击左栏任一科目
- Then 右栏应立即切换为该科目详情，且无需继续向下滚动才能看到结果

### 10.3 单科内容定位

- Given 用户已进入某一科目详情
- When 用户输入章节名或知识点关键字
- Then 页面应高亮匹配结果，并展开到对应章节分支

### 10.4 上下文稳定性

- Given 用户当前正在查看某一科目
- When 用户修改筛选条件
- Then 若该科目仍满足筛选，页面不得自动切换到其他科目

### 10.5 错误态清晰

- Given 接口请求失败
- When 页面无法获取大纲数据
- Then 页面应展示错误态和重试动作，而不是仅用消息提示或表现为普通空态

### 10.6 移动端可阅读

- Given 用户在移动端进入考试大纲页
- When 页面加载完成
- Then 用户应先看到当前科目详情主内容，且能通过抽屉或折叠区切换其他科目

### 10.7 版本可信度

- Given 用户查看某一科目大纲
- When 详情头部渲染完成
- Then 页面应展示版本号或政策版本、生成时间等至少一种来源凭据

## 11. 实施优先级

### P0

- 调整桌面端为左目录右详情双栏结构
- 压缩顶部概览卡为轻量摘要
- 补页面级错误态
- 修复筛选变更导致的静默跳科目问题

### P1

- 新增单科大纲内容搜索
- 新增章节导航与锚点跳转
- 新增展开全部 / 收起全部 / 回到顶部

### P2

- 优化移动端目录抽屉体验
- 补充大纲来源文件与版本展示细节
- 评估是否增加最近查看科目记忆能力

## 12. 进入开发前需确认项

以下内容在进入实现前应视为已冻结：

1. 桌面端确定采用“左目录右详情”，而非继续上下结构。
2. 单科内搜索优先采用前端本地检索，不新增写接口。
3. 脑图默认采用渐进展开，而非继续全展开。
4. 页面必须具备加载态、空态、错误态三类明确语义。
5. 页面保留全科总览定位，不跟随当前练习科目自动切换。

## 13. 下一步建议

- 先按本方案出低保真结构稿或直接进入前端实现。
- 实现阶段进入 `$fullstack-unified-development-standards`，按本文件作为开发基线推进。
