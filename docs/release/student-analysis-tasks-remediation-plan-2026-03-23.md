本方案聚焦学生端 `知识诊断 > 今日任务` 子页：[`/student/analysis/tasks`](/Users/shaotongli/Documents/newaitiku/frontend/src/views/Student/Tasks.vue)

## 2. 整改目标

本次整改不是继续堆卡片，也不是把“今日任务”做成另一个分析页，而是把它收敛为真正的“诊断后的执行页”。

整改后页面应满足以下目标：

1. 学生进入后 3 秒内看懂“今天先做什么”。
2. 切换科目后，页面中的任务、推荐动作、弱项和记录必须口径一致。
3. 页面要能明确区分“没数据”和“加载失败”。
4. 首屏以执行任务为主，不被后台统计和次级指标抢走注意力。
5. 所有动作按钮的名称、落点和用户预期一致。

## 3. 页面职责冻结

### 3.1 页面定位

`知识诊断 > 今日任务` 是学生端“先诊断、后执行”链路中的执行子页，和 `知识诊断 > 练习积分` 一起构成知识诊断下的执行/详情二级结构。

它负责：

- 承接诊断后的当天主动作
- 展示今日任务卡片流
- 展示失败/待处理任务的恢复入口
- 展示最近执行记录
- 提供去刷题升本、去错题修复、去补弱项的直达动作

`知识诊断 > 练习积分` 负责：

- 作为积分详情页承接首页里的“刷题段位摘要”
- 展示同科目排名、段位进度、分享入口等详情能力
- 不再作为学生端一级导航单独存在

它不负责：

- 承担知识结构全景分析
- 承担系统任务中心的后台统计看板
- 承担高权重 AI 配额或运行状态展示
- 把图谱类信息作为首屏主视觉

### 3.2 首屏优先级

首屏信息顺序冻结为：

1. 今日推荐动作
2. 今日任务列表
3. 失败/待处理任务
4. 最近执行记录
5. 侧栏中的轻量进度与弱项提醒

以下信息不得高于任务动作：

- 路径覆盖率
- AI 剩余配额
- 后台任务计数摘要

## 4. 当前问题清单

### 4.1 P0 问题

1. 切换科目后只刷新弱项图谱，不刷新任务和推荐动作，容易造成跨科目错配。
2. 核心接口失败后页面退化成空状态，用户无法判断是“暂无数据”还是“加载失败”。
3. 首屏仍保留较重的分析型卡片，执行主线不够突出。

### 4.2 P1 问题

1. 顶部任务统计使用的是当前分页记录，不是全量统计，展示口径有误导风险。
2. `查看结果 / 重试任务 / 再次进入` 等按钮文案与实际跳转不完全一致。
3. 图谱相关模块有加载状态变量，但页面上没有真实的加载反馈。

### 4.3 P2 问题

1. 空状态下的按钮组合重复，主次关系不够明确。
2. “失败任务重试卡”当前识别度不足，尚未成为页面中的独立重点模块。
3. 局部文案仍带有偏系统态、偏后台态的语气。

## 6. 具体整改方案

### 6.1 P0：先修正错误口径和主流程

#### P0-1 科目切换一致性

- 切换 `subjectCode` 后，必须同时刷新：
  - `dashboard`
  - 今日任务
  - 最近执行记录
  - 弱项摘要
- 页面刷新期间，主内容区进入明确的加载态，避免旧科目数据残留。

#### P0-2 页内错误态

- 为以下模块分别补齐错误态卡片：
  - 今日任务
  - 最近执行记录
  - 弱项提醒
- 错误态必须包含：
  - 错误提示
  - 重试按钮
  - 兜底跳转动作
- 不允许仅通过 toast 告知失败后直接渲染为空页面。

#### P0-3 首屏结构收敛

- 首屏保留一个执行 Hero：
  - 当前推荐动作
  - 当前行动原因
  - 主按钮
  - 次按钮
- 下方直接进入“今日任务卡片流”。
- “路径覆盖率 / AI 配额 / 任务统计”下移到页面底部，或压缩为次级摘要，不进入首屏主视觉。

### 6.2 P1：优化理解成本和动作闭环

#### P1-1 任务统计口径修正

- 如果后端能提供全量统计，则前端展示真实总数。
- 如果当前只能拿到分页记录，则文案必须改为：
  - `最近任务`
  - `最近 20 条中进行中`
  - `最近 20 条中失败`
- 禁止使用容易被理解成全量任务池的数据标题。

#### P1-2 CTA 语义统一

- `查看结果` 只能用于能真正进入结果页的任务。
- `重试任务` 只能用于失败任务，并优先命中失败恢复路径。
- `继续执行` 只能用于进行中任务。
- `再次进入` 只用于已完成但允许再次练习的任务。
- 若没有明确结果页或恢复页，按钮文案应改为更真实的行为描述。

#### P1-3 图谱相关状态补齐

- 弱项提醒和路径覆盖率区块应接入独立 loading。
- loading 期间显示骨架或占位卡，不直接显示 `0%` 或空列表。

### 6.3 P2：提升可读性和验收稳定性

#### P2-1 空状态动作精简

- 今日任务为空时：
  - 主按钮：`去刷题升本`
  - 次按钮：`查看诊断总览`
- 最近记录为空时：
  - 主按钮：`先完成今日打卡` 或 `去刷题升本`
- 弱项为空时：
  - 主按钮：`查看诊断总览`

#### P2-2 失败任务强化

- 把失败任务从普通记录卡中抽出来，形成单独“待恢复任务”区域。
- 失败任务卡需包含：
  - 失败原因摘要
  - 最近更新时间
  - 主恢复按钮

#### P2-3 文案基线统一

- 保持 `知识诊断今日任务` 命名不变。
- 页面语义统一为：
  - 诊断后的执行动作
  - 当天主线
  - 今日任务
  - 最近执行记录
- 避免再次出现旧“独立任务中心”口径。

## 8. Given / When / Then 验收标准

### 8.1 首屏执行性

- Given 学生进入 `知识诊断 > 今日任务`
- When 页面加载完成
- Then 首屏必须先展示一个明确的“当前推荐动作”
- Then 首屏必须能直接看到今日任务列表
- Then 首屏不应先出现高权重系统统计卡片

### 8.2 科目切换一致性

- Given 学生在今日任务页切换当前科目
- When 新科目数据加载完成
- Then 弱项提醒等诊断信息应切换到当前科目
- Then 今日任务与最近执行记录允许继续展示全局汇总数据
- Then 页面中不应出现“任务已按当前科目过滤”的误导性表达

### 8.3 错误态可识别

- Given 今日任务、任务记录或弱项接口任一加载失败
- When 页面展示反馈
- Then 用户必须能看到页内错误提示
- Then 用户必须能看到重试动作
- Then 页面不得把错误直接渲染成“暂无数据”

### 8.4 空状态可行动

- Given 当前账号暂无今日任务
- When 页面加载完成
- Then 页面必须展示空状态说明
- Then 页面必须至少提供 `去刷题升本` 或 `查看诊断总览` 之一作为主动作

### 8.5 统计口径清晰

- Given 页面展示任务统计摘要
- When 用户查看统计标题和数字
- Then 用户必须能明确知道该数字是“全量统计”还是“最近记录统计”
- Then 页面不得使用会让用户误认为是全量统计的模糊标题

### 8.6 CTA 语义一致

- Given 页面上出现 `查看结果 / 重试任务 / 继续执行 / 再次进入`
- When 用户点击对应按钮
- Then 实际跳转落点必须与按钮文案语义一致
- Then 不允许出现“查看结果”却落到诊断总览的情况

## 9. UAT 场景

1. 学生进入 `知识诊断 > 今日任务`，确认首屏先看到推荐动作和任务卡片流。
2. 切换到另一个科目，确认页面所有模块数据同步刷新且口径一致。
3. 模拟任务接口失败，确认页内展示错误态和重试按钮。
4. 用无今日任务账号进入页面，确认空状态主动作清晰。
5. 点击失败任务和已完成任务的按钮，确认文案与落点一致。
6. 检查统计卡标题，确认不存在“最近 20 条”冒充“全部任务”的情况。

## 10. 进入开发前的冻结项

进入开发前，需要再明确以下三点：

1. `任务统计` 是显示全量数据，还是只显示最近记录数据。
2. `查看结果` 的标准落点是什么页面。
3. 已冻结：科目切换时，今日任务与任务记录允许跨科目汇总展示，不按当前科目过滤。

若以上三项没有明确结论，则视为方案未冻结，不建议直接进入实现。
