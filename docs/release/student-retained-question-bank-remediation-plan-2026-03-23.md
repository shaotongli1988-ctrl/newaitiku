# 沉淀题库整改工作计划 v1（2026-03-23）

- 三阶段路由结果：`开发前（readiness）`
- 对应总技能：`software-development-readiness-governance`
- 当前结论：允许进入整改方案冻结与开发排期评审，不建议在状态模型、接口合同和复习计划正式能力未冻结前直接分散开改。

本方案聚焦学生端 `我的题库 > 沉淀题库`，覆盖以下范围：

- 学生端沉淀题库页面与交互

- 开发前守卫脚本已收敛到 `docs/skills/software-development-readiness-governance/scripts/software_development_readiness_guard.py`，本方案按当前技能镜像与仓库文档模板冻结。

## 2. 整改目标

本次整改不是继续优化页面样式，也不是只补几个接口参数，而是把沉淀题库从“可用但治理未收口”升级为“模型、接口、页面、测试一致的正式能力”。

整改后应满足以下目标：

1. 沉淀题库的收藏、归档、恢复、复习、导出统一基于正式表，不再依赖 `extJson` 热状态作为主读写路径。
2. 复习计划具备正式数据模型、正式接口和可追踪执行状态，不再是运行时临时计算结果。
3. 列表、汇总、导出、复习计划、在库作答使用同一套筛选口径和状态真相源。
4. 页面、接口、文档、测试统一使用“沉淀题库”语义，不再长期并存 `personal-bank` 双命名。
5. 沉淀题库主链路具备完整自动化证据，至少覆盖“列表 -> 汇总 -> 在库作答 -> 导出 -> 恢复/计划执行”闭环。

### 3.3 非协商规则

以下规则在本次整改中不接受继续保留：

1. `extJson` 继续承载沉淀题库高频热状态作为主读路径。
2. 复习计划继续以 `summary.reviewPlan` 的临时分组方式存在。
3. 新功能继续建立在 `personal-bank` 历史语义之上。
4. 列表、汇总、导出各自维护不同筛选逻辑。
5. 沉淀题库正式能力无完整 e2e 证据即进入提测。

## 4. 产品定义冻结

### 4.1 能力定位

`沉淀题库` 不是普通收藏夹，也不是错题中心的别名。

它是学生个人学习资产池，负责承接：

- 已收藏待练题
- 已斩获归档题
- 可继续回练题
- 正式复习计划中的题目
- 沉淀题目的导出与恢复

### 4.2 与其他页面的职责边界

`沉淀题库` 负责：

- 搜索与筛选沉淀题
- 承接正式复习计划
- 提供在库作答入口
- 提供导出与恢复入口
- 展示沉淀题的必要上下文与进度

`沉淀题库` 不负责：

- 替代知识诊断做全景分析
- 承担错题中心的主修复流程
- 承担系统级任务中心职责
- 以“临时推荐分组”代替正式计划

### 4.3 状态机冻结

沉淀题库相关状态至少冻结为以下集合：

- `COLLECTED`：已进入沉淀题库
- `ARCHIVED`：已斩获归档
- `RESTORED`：已恢复到错题中心
- `REVIEW_DUE`：存在待执行复习
- `REVIEWING`：复习计划执行中
- `REVIEW_COMPLETED`：当前计划已完成

补充规则：

- “收藏进入沉淀题库”与“错题已斩获归档”是两个不同来源，不允许继续只靠 `sourceLabel` 文案判断。
- “恢复到错题中心”必须留下正式状态与时间戳，而不是仅改 JSON 标志位。
- 复习计划状态必须独立于沉淀题静态状态存在。

## 5. 当前问题清单

### 5.1 P0 问题

1. 沉淀题库热状态仍主要通过 `record_ext.personalBank / wrongBook` 读写，未真正收敛到正式表主路径。
2. 复习计划仍由汇总逻辑运行时临时拼装，未形成正式产品模型。
3. 沉淀题库列表、汇总、导出未共享统一仓储层主查询，存在口径和性能扩散风险。
4. 查询链路存在逐题读取学生记录的 N+1 倾向，题量上来后性能风险高。
5. `sourceType` 等筛选字段真实归属不清，存在接口参数可传但过滤结果不可靠的漂移风险。

### 5.2 P1 问题

1. 页面、接口、文档、导出仍混用“沉淀题库 / personal-bank / 个人题库”三套语义。
2. 复习计划目前更像建议分组，而不是具备生命周期、执行记录和完成状态的正式能力。
3. 前端已做搜索驱动收口，但后端模型与正式合同未同步冻结，形成“页面先行、建模滞后”。

### 5.3 P2 问题

1. 当前自动化以 API 验证为主，沉淀题库主链路缺少完整真点击闭环。
2. 相关文档已经收敛为正式页面语义，但实施清单、接口合同和测试矩阵未完全同步。
3. 导出文案、返回结构与命名仍保留历史痕迹，增加后续对外说明成本。

## 6. 方案冻结

### 6.1 数据模型方案

本次整改冻结以下原则：

1. `student_question_record` 作为沉淀题库状态真相源。
2. 沉淀题库高频状态必须进入正式列，不继续以 `extJson.personalBank / wrongBook` 作为主读写依据。
3. 复习计划必须新增正式表，不再挂在 `summary` 或页面本地状态中。
4. 题目表 `question` 仅承接题目静态内容与低频扩展，不承接学生学习热状态。

建议新增或明确以下正式结构：

- `student_question_record`
  - 沉淀状态
  - 归档状态
  - 恢复状态
  - 相关时间戳
  - 来源类型
  - 最近复习状态
- `student_review_plan`
  - `id`
  - `studentUserId`
  - `planType`
  - `planName`
  - `status`
  - `generatedAt`
  - `startedAt`
  - `completedAt`
  - `lastExecutedAt`
  - `extJson`
- `student_review_plan_item`
  - `id`
  - `planId`
  - `questionId`
  - `status`
  - `sort`
  - `completedAt`
  - `extJson`

### 6.2 查询链路方案

统一查询顺序冻结为：

1. 先按 `student_question_record / student_review_plan / student_review_plan_item` 过滤学生态与计划态。
2. 再按 `question` 关联题目静态字段。
3. 列表、汇总、导出共用同一套过滤器定义。
4. 所有分页、统计、导出均基于同一查询口径。

禁止继续采用：

- 先拉全量题目，再逐题查学生记录
- 导出重新走一套独立过滤逻辑
- 汇总只基于前端页内结果做二次统计

### 6.3 复习计划正式化方案

复习计划冻结为正式能力，不再以“建议分组”定义。

至少需要明确：

1. 计划类型：
   - 今日优先复习
   - 高频易错加练
   - 未作答补齐
   - 后续允许扩展自定义计划
2. 计划生成规则：
   - 自动生成条件
   - 重复生成覆盖规则
   - 历史计划保留规则
3. 计划执行状态：
   - 未开始
   - 执行中
   - 已完成
   - 已失效
4. 计划闭环：
   - 进入计划
   - 执行题目
   - 题目完成记录
   - 计划完成判断

### 6.4 命名与语义方案

统一命名冻结为：

- 页面：`沉淀题库`
- 正式能力：`沉淀题库`
- 文档与验收：`沉淀题库`
- 导出标题：`沉淀题库导出`

历史 `personal-bank` 不再作为长期兼容目标，仅允许作为代码迁移期内部替换对象，不允许继续扩散到新接口和新文档。

## 9. Given / When / Then 验收标准

### 9.1 状态真相源

- Given 学生收藏、归档、恢复、执行复习计划
- When 系统写入与读取沉淀题库状态
- Then 主读写路径必须基于正式表
- Then 不得再依赖 `extJson` 热状态作为主口径

### 9.2 复习计划正式化

- Given 学生进入沉淀题库
- When 页面加载复习计划
- Then 计划必须来自正式计划接口
- Then 计划必须具备正式状态、题量和执行进度
- Then 不得仅由汇总接口临时拼装计划结果

### 9.3 查询口径一致

- Given 学生在沉淀题库设置筛选条件
- When 查看列表、汇总和导出
- Then 三者结果必须使用同一套筛选口径
- Then 不得出现“列表结果”和“导出结果”不一致

### 9.4 命名统一

- Given 学生使用沉淀题库正式功能
- When 查看页面、导出结果、文档和测试基线
- Then 对外语义必须统一为“沉淀题库”
- Then 不得继续把 `personal-bank` 作为正式产品语义暴露

### 9.5 自动化覆盖

- Given 沉淀题库进入提测
- When 执行最小回归
- Then 必须至少覆盖“列表 -> 汇总 -> 在库作答 -> 导出 -> 恢复/计划执行”完整链路

## 10. 当前判断

- 当前沉淀题库具备基础可用性，但尚未达到“模型、计划、接口、测试全面收口”的正式交付标准。
- 当前最关键阻断项不是页面样式，而是：
  - 状态真相源未完全收口
  - 复习计划未正式化
  - 查询链路未统一
  - 自动化证据不完整

在以下条件全部满足前，不建议将本能力视为“整改完成”：

1. 复习计划正式表与正式接口完成落版
2. 沉淀题库主读写路径完成切换
3. 命名和文档完成统一
4. 主链路 e2e 已补齐
