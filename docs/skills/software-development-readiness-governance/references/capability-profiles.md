# 能力特性细则（软件开发前准备）

本文件定义开发前准备总技能的核心能力面，统一由总守卫调度。

## A. 需求冻结特性

- 需求范围、版本边界、非目标项、依赖前提必须明确。
- 未冻结需求不进入设计，未明确设计不进入编码。
- 新增口头变更必须重新评估影响面。

## B. 验收口径特性

- 功能、异常、边界、权限、状态都必须具备验收条件。
- 验收标准应能直接映射到测试与 UAT 场景。
- “大概是这样”不应作为开工依据。

## C. 契约字段特性

- 字段名、类型、默认值、是否必填、枚举值、错误结构必须统一。
- 接口响应包、分页结构、时间与 ID 语义必须先定。
- 不应带着“后面再统一”的字段歧义进入开发。

## D. 数据建模特性

- 表结构、实体关系、唯一性、索引方向、历史兼容策略必须先对齐。
- 业务字段语义与数据库字段语义必须一致。
- 不应让“数据库怎么落”在开发中途边写边猜。

## E. 权限与作用域特性

- 角色、权限点、菜单权限、按钮权限、接口鉴权必须明确。
- 数据作用域应明确到租户、学校、年级、班级、个人等层级。
- “看得见但不该改”与“改得到但不该看”都应在开工前识别。

## F. 业务边界与状态流特性

- 模块责任、跨模块写入边界、事件归属、状态流转规则必须明确。
- 同一业务规则只能有一个权威归属点。
- 不应在开发中途发现多个模块都在偷偷改同一个状态。

## G. 视觉与交互标准特性

- 页面骨架、表单/表格规范、按钮层级、反馈策略必须统一。
- 加载态、空态、错误态、禁用态、确认态应有标准答案。
- 多端页面不应靠各自理解自由发挥。

## H. 校验与错误语义特性

- 前后端校验规则、提示文案、错误码命名空间、异常语义必须一致。
- 需要明确哪些错误可重试、哪些错误应中断、哪些错误可提示用户修复。
- 不应让前端和后端各自发明一套错误话术。

## I. 安全与非功能基线特性

- 鉴权、输入校验、最小权限、敏感日志、审计要求在开发前明确。
- 性能、容量、日志、埋点、监控、审计等非功能基线应在关键场景先定义。
- 不因“先把功能做出来”而把安全与可观测性推迟到上线前。

## J. 协作与方案特性

- 技术方案、架构约束、是否允许引新依赖、分支策略、联调顺序必须明确。
- 没有决策记录的重大技术选择，应视为开发前准备不足。
- 开工前要明确“谁负责什么、如何合并、何时联调、何时回归”。

## K. 模式补充

- `questionBank` 模式额外要求模块白名单、固定响应包、`extJson` 扩展与 ID 关联规则一致。
- `global` 模式遵循当前项目全局契约，但仍需完成字段、权限、状态、视觉与错误语义的统一。
