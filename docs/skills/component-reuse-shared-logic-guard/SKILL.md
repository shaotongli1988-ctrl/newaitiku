# 组件复用与公共逻辑提取守卫

## 定位

- 本技能是 `fullstack-unified-development-standards` 的子技能，不单独替代主技能。
- 本技能只负责“实现层收敛”，不接管接口契约、权限、状态机、错误码、测试矩阵、交付文档的主责。
- 目标是把“代码能跑”进一步收敛为“实现方式统一、可复用、可维护、可迁移”。

## 适用场景

- 新增页面、列表页、表单页、弹窗页、抽屉页。
- 同类查询、分页、重置、弹窗控制、表单提交逻辑重复出现。
- 页面内直接写 `axios/fetch`、时间格式各写一套、分页结构各页自定义。
- 后端新增实体、分页入参、返回体、公共异常/日志/校验逻辑时需要统一抽象。
- 联调前、批量重构后、上线前，需要额外检查“实现层是否已经收敛”。

## 冲突优先级

1. 用户在当前任务中的明确要求
2. `questionBank` 固定契约
3. `fullstack-unified-development-standards` 主技能规则
4. 本技能实现层收敛规则
5. 项目局部历史写法

## 核心检查面

1. 重复代码是否已抽为公共组件、公共 Hook、公共函数。
2. 页面请求是否统一走 `frontend/src/api/request.js` 或项目统一 `request` 封装。
3. 公共组件是否支持 `props`、事件、默认值、插槽，且不写死业务逻辑。
4. 分页结构、返回体结构、时间格式、`extJson` 结构是否统一。
5. 公共工具函数是否保持纯函数、无副作用、可单测。
6. 后端实体、分页参数、统一返回、异常处理、校验、日志是否优先复用公共基类或公共模块。
7. 命名、常量、配置、文本是否已收敛，避免页面内硬编码。

## 默认规则

- 同构重复实现出现 2 次时，至少输出抽取建议或补齐迁移清单。
- 同构重复实现出现 3 次及以上，或已形成明显复制粘贴链路时，应优先抽取公共层。
- 不允许在旧坏实现外再包一层兼容胶水、适配器或额外分支来“少改一点”。
- 已有更合理的公共层、组件、Hook、工具或请求封装时，应直接迁移并替换旧实现，不得继续新旧并挂。
- 前端页面禁止直接写裸 `axios`；静态页存量局部 `request` 不应作为常驻例外继续保留。
- 仅明确过渡窗口允许短期保留局部 `request`，且必须写明负责人、截止时间、替换目标并进入迁移清单。
- 分页结构固定为 `{ items, page, size, total }`，接口响应固定为 `{ code, message, data }`。
- `extJson` 必须保持对象语义，不得在新增实现中退回字符串承载扩展字段。
- 后端新增表优先继承 `BaseEntity` 语义：`id`、`createTime`、`updateTime`、`isDelete`、`remark`、`extJson`。
- 存量核心表若尚未完全补齐 `isDelete/remark`，按迁移目标治理，不阻断非本次任务范围。

## 执行方式

接入主技能后，推荐按以下阶段执行：

1. `start`
   - 判断本次需求是否涉及组件复用、公共逻辑、公共模块或重复实现收敛。
   - 冻结本次改动允许新增的公共层边界，避免边做边扩散。
2. `batch`
   - 扫描当前批次是否引入新的重复实现、页面直写请求、结构不统一。
   - 对未抽取项给出 `high/medium/low` 风险分级与修复方向。
3. `final`
   - 确认高风险实现漂移已收敛。
   - 仍保留的存量问题必须列入迁移清单，不能静默跳过。

在守卫脚本落地前，按 [references/reuse-extraction-checklist.md](references/reuse-extraction-checklist.md) 手动执行。
如已接入脚本，优先输出 `--report-md` 和 `--report-json`，报告中应包含摘要、发现清单与迁移清单。

## 风险分级

- `high`
  - 同构代码已重复 3 次及以上仍继续复制
  - 页面新增接口仍直接写裸 `axios/fetch`
  - 同一模块混用不同分页结构或返回体结构
  - 公共组件写死业务分支，导致其他页面无法复用
- `medium`
  - 同构逻辑重复 2 次但尚未抽取
  - 存在可抽取的通用 Hook/工具但仍散落在页面中
  - 存量静态页仍使用局部 `request`，但迁移清单未更新
- `low`
  - 命名、注释、常量提取仍有零散不一致
  - 可读性或抽象层级有优化空间，但暂不阻断交付

## 修复协议

1. 先收敛 `high`，再处理 `medium`，最后清理 `low`。
2. 优先复用已有公共层；不存在时，先补公共层再写业务代码。
3. 抽公共层时保持“无业务写死、可配置、可扩展、可测试”。
4. 发现旧实现方向错误或明显落后于统一方案时，优先替换旧实现，不做补丁式兼容叠加。
5. 如抽取会扩大到多个业务域或多个题库模块，暂停并确认授权边界。
6. 存量历史代码无法在本次完全迁移时，必须补迁移清单和后续收敛点；仅明确过渡窗口允许临时兼容，且必须带截止时间。
