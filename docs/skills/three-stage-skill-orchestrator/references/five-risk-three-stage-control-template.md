# 五类高风险问题三阶段控制点模板

## 目的

- 为三阶段体系提供一份共享口径，避免同一类风险在开发前、开发中、开发后各写一套说法。
- 以下五类问题默认使用本模板收敛：
  - 高并发、超卖、重复下单
  - 数据错误、事务不对
  - 慢查询、数据库崩掉
  - 部署不上、线上环境炸
  - 安全漏洞、越权

## 使用规则

1. 开发前总技能使用“开发前控制点”定义方案、边界和验收口径。
2. 开发中总技能使用“开发中控制点”把方案落地为代码、配置、文档和测试。
3. 开发后总技能使用“开发后控制点”检查前两阶段的承诺是否兑现。
4. 任一阶段如果缺失本阶段必需输出，应明确阻断，不得静默跳过。

## 一、高并发、超卖、重复下单

### 开发前控制点

- 明确并发模型：热点对象、并发峰值、库存扣减方式、是否允许排队或预占。
- 明确幂等策略：幂等键来源、幂等窗口、重复请求返回语义。
- 明确防超卖策略：乐观锁、悲观锁、库存预占、消息串行化或其他等价方案。
- 明确流量保护策略：限流、降级、隔离、熔断、预热。
- 明确验收标准：压测目标、错误率、库存一致性、重复下单预期行为。

### 开发中控制点

- 落地幂等、防重、库存一致性代码路径与异常分支。
- 落地热点隔离、限流降级、缓存或队列等并发保护机制。
- 补齐关键链路日志、指标和告警字段。
- 补齐并发、重复提交、库存边界的测试或压测证据。

### 开发后控制点

- 检查是否存在幂等证据、防重证据、库存一致性证据。
- 检查压测结果、限流降级配置、热点监控是否齐全。
- 对“可能超卖 / 可能重复下单 / 高峰不可控”给出阻断结论。

## 二、数据错误、事务不对

### 开发前控制点

- 明确事务边界：哪些操作必须同事务，哪些操作允许异步补偿。
- 明确状态流与回滚语义：成功、失败、取消、超时、重试的状态规则。
- 明确一致性要求：强一致、最终一致或人工兜底。
- 明确异常与补偿策略：重试、补偿、对账、人工修复。
- 明确验收标准：状态一致性、脏数据回放、失败回滚预期。

### 开发中控制点

- 落地事务、状态机、异常回滚与补偿逻辑。
- 对齐数据库模型、接口契约、服务层语义和错误码。
- 补齐正常、异常、边界、回滚和补偿测试。
- 补齐关键数据修复或对账入口说明。

### 开发后控制点

- 检查事务边界、状态流和补偿逻辑是否与设计一致。
- 检查接口、数据库、前端状态展示是否一致。
- 对“会产生脏数据 / 回滚不完整 / 状态错乱”给出阻断结论。

## 三、慢查询、数据库崩掉

### 开发前控制点

- 明确核心查询路径、数据规模、峰值访问量和热点对象。
- 明确索引、分页、批量化、缓存和读写分离等策略。
- 明确大表 DDL、数据回填、迁移顺序与回滚方案。
- 明确容量假设、性能目标与告警阈值。

### 开发中控制点

- 落地索引、分页、批量查询、缓存与 SQL 优化。
- 避免 N+1、全表扫描、无边界大查询和高风险 DDL 直推。
- 补齐性能验证、基准测试或 explain 证据。
- 补齐数据库监控、慢日志和容量相关观测项。

### 开发后控制点

- 检查慢 SQL、N+1、索引缺失、热点 SQL、容量缺口是否仍存在。
- 检查迁移顺序、回填方案和回滚脚本是否齐全。
- 对“可能打爆数据库 / 慢查询不可控 / 迁移高风险”给出阻断结论。

## 四、部署不上、线上环境炸

### 开发前控制点

- 明确环境矩阵：dev/test/prod 的配置差异、域名、证书、回调地址。
- 明确发布顺序：代码、配置、数据库脚本、开关、依赖的先后关系。
- 明确回滚策略：应用、配置、数据库三层的回退路径。
- 明确监控、日志、告警和止血要求。

### 开发中控制点

- 落地配置解耦、环境变量模板、脚本版本匹配和部署文档。
- 落地监控埋点、日志字段、告警规则和故障处置入口。
- 补齐回滚步骤、止血动作、运行手册。
- 补齐多环境验证与发布演练证据。

### 开发后控制点

- 检查构建产物、配置矩阵、数据库脚本、监控告警、回滚条件是否齐全。
- 检查是否存在“代码能发但环境不能发”的缺口。
- 对“无法部署 / 无法回滚 / 出事不可止血”给出阻断结论。

## 五、安全漏洞、越权

### 开发前控制点

- 明确角色、权限点、资源边界和最小权限原则。
- 明确接口鉴权、前端显隐、输入校验、输出转义要求。
- 明确敏感数据范围、日志脱敏和密钥管理规则。
- 明确安全验收标准：越权、注入、XSS/CSRF、敏感泄漏等测试范围。

### 开发中控制点

- 落地后端鉴权、前端权限显隐、输入校验、输出转义与脱敏处理。
- 落地权限键对齐、RBAC 映射与异常响应语义。
- 补齐权限回归、安全用例和敏感信息扫描证据。

### 开发后控制点

- 检查越权访问、漏鉴权、注入、XSS/CSRF、敏感泄漏风险是否已关闭。
- 检查 RBAC、前后端权限键和日志脱敏是否一致。
- 对“存在高风险漏洞 / 存在越权路径 / 存在泄密风险”给出阻断结论。

## 输出模板

每次引用本模板时，至少输出：

- 当前阶段
- 命中的风险类型
- 本阶段必做控制点
- 已有证据
- 缺失项
- 是否允许进入下一阶段

## 阻断规则

- 开发前缺方案、边界或验收口径：不得进入开发。
- 开发中缺真实实现、测试或文档证据：不得宣称已完成。
- 开发后缺验收证据、回滚条件或安全证明：不得进入提测或上线。

## 数据建模强约束

以下规则默认适用于三阶段，不作为“建议项”，而作为需要明确遵守或明确豁免的约束：

- `extJson` 不再承载高频业务状态。
- 任何需要并发写安全的状态必须入表。
- 任何需要幂等的写接口必须有数据库唯一约束。
- 任何报告型数据如果要分页、统计、筛选，不能只存 JSON。

阶段解释：

1. 开发前：必须先判断当前字段/状态是“低频扩展字段”还是“高频业务状态”，并给出入表或保留 `extJson` 的依据。
2. 开发中：如果命中上述四条任一约束，必须优先调整表结构、索引、唯一键、Repository 写法和事务边界，不允许仅通过补判断逻辑规避。
3. 开发后：必须检查是否仍有高频热状态留在 `extJson`、是否仍有幂等接口仅靠代码判断、是否仍有报告数据仅存在 JSON。
