## 3. 数据模型与字段标准

### 3.1 数据库主合同（DDL）

- `user`：`id, phone, password, status, extJson, createTime, updateTime`
- `userAuth`：`id, userId, type, openid, unionid, extJson, createTime, updateTime`
- `knowledge`：`id, parentId, name, sort, status, extJson, createTime, updateTime`
- `question`：`id, knowledgeId, userId, type, stem, optionsJson, answer, status, extJson, createTime, updateTime`
- `task`：`id, userId, type, status, progress, extJson, createTime, updateTime`

### 3.2 关系与扩展边界

- 关系字段采用 ID 关联（如 `knowledgeId`、`userId`、`task.userId`）。
- `question.extJson` 承接题目扩展：内容体系、难度、知识标签、审核轨迹、组卷绑定等。
- 学生题目态已从 `user.extJson.studentRecords[questionId]` 收口到 `student_question_record`。
- 系统设置/消息/模板承接于系统用户 `__system__` 的 `user.extJson`；其中试卷报告、消息发送历史已收口到 `paper_report`、`message_send_history`。
- 异步任务结果承接于 `task.extJson`。

### 3.3 历史表处理

- DDL 启动时会显式 `DROP` 历史表/旧表名：`subjects`、`questions`、`question_reviews`、`student_question_bank`、`question_vectors`。
- 当前仓储层读写已统一落在五表合同，未扫描到对上述历史表的存量读写 SQL。

## 6. 状态标准

### 6.1 题目状态流转

- `DRAFT -> QA_IN_PROGRESS | REVIEW_PENDING | REJECTED`
- `QA_IN_PROGRESS -> REVIEW_PENDING | REJECTED`
- `REVIEW_PENDING -> PUBLISHED | REJECTED`
- `REJECTED -> DRAFT`
- `PUBLISHED` 无后续流转

角色约束补充：

- 草稿提交流转通常要求题目拥有者执行。
- `QA_IN_PROGRESS -> REVIEW_PENDING` 不允许题目拥有者自审通过。
- `REVIEW_PENDING -> PUBLISHED/REJECTED` 要求非题目拥有者教师终审。

### 6.2 试卷状态流转

- 允许状态：`DRAFT`、`REVIEW_PENDING`、`PUBLISHED`、`OFFLINE`
- 流转规则：
  - `DRAFT -> REVIEW_PENDING`
  - `REVIEW_PENDING -> DRAFT | PUBLISHED`
  - `PUBLISHED -> OFFLINE`
  - `OFFLINE -> REVIEW_PENDING`

### 6.3 任务状态流转

- 状态集合：`QUEUED`、`RUNNING`、`COMPLETED`、`FAILED`、`CANCELLED`
- 终态：`COMPLETED`、`FAILED`、`CANCELLED`（终态后不可重复执行取消/推进）

## 7. 权限标准

### 7.1 后端守卫链路

- 鉴权主体：`Actor(role, user_id, assigned_joint_group_code)`
- Token/会话来源：`Authorization Bearer` 优先，回退 `qbAccessToken` Cookie
- 超管写接口 CSRF 双提交：`X-CSRF-Token` 与 `qbCsrfToken` 一致
- 页面角色不匹配时 302 重定向到各角色首页
- `admin/*` API 强制 `super_admin`
- 业务域守卫函数：
  - `require_question_operator`
  - `require_paper_operator`
  - `require_analytics_operator`
  - `require_student`
  - `require_super_admin`

### 7.2 前端权限显隐

- Vue 路由 `meta.requiredPermissions` 控制页面级访问。
- Vue 指令 `v-permission` 支持 `all/any` 表达式。
- 前端权限状态统一由 `frontend/src/stores/userStore.js` 驱动，并通过 `/api/question-bank/auth/me` 拉取当前权限集合。

## 8. 校验标准

后端模型校验（Pydantic）：

- 全局 `extra="forbid"`，拒绝未定义字段。
- 题目校验：
  - `type` 必须在题型枚举内
  - `status` 必须在题目状态枚举内
  - `optionsJson` 必须是合法 JSON，客观题必须非空数组
  - `extJson` 必须是 JSON 对象
  - `extJson.difficulty` 仅 `easy|medium|hard`
  - `extJson.knowledgeTags` 限 1-3
- 知识点校验：
  - `status` 限 `ENABLED|DISABLED`
  - `name` 非空
  - `extJson` 必须对象
- 题目创建/更新新合同模型（`app/contracts.py`）补充了 snake/camel alias 与内容体系范围校验。

前端请求层校验：

- 请求统一经 `frontend/src/api/services/questionBank.js` 归一校验后发出。
- 对关键请求参数进行 `required`、长度、范围与类型约束。
- 禁止 camel 别名直接透传（强制字段名契约）。

## 9. 错误标准

- 统一异常类型：`QuestionBankError(code, message, status_code)`。
- Question 命名空间错误码：
  - `QUESTION_NOT_FOUND`
  - `QUESTION_FORBIDDEN`
  - `QUESTION_VALIDATION_FAILED`
  - `QUESTION_INVALID_STATUS`
  - `QUESTION_DATABASE_ERROR`
- Task 命名空间错误码：
  - `TASK_NOT_FOUND`
  - `TASK_FORBIDDEN`
  - `TASK_VALIDATION_FAILED`
  - `TASK_INVALID_STATUS`
  - `TASK_DATABASE_ERROR`
- 异常返回统一 envelope：`{ code, message, data: null }`。
