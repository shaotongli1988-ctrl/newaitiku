## 2) 决策 SLA 规则

| 严重级别 | 响应时限 | 决策时限 | 逾期升级路径 |
| --- | --- | --- | --- |
| P0 (阻断发布/核心链路不可用) | 15 分钟 | 2 小时 | 项目 Owner -> 技术负责人 -> 发布负责人 |
| P1 (影响主流程但可临时绕行) | 2 小时 | 1 个工作日 | 项目 Owner -> 模块负责人 |
| P2 (非主流程/优化类阻塞) | 1 个工作日 | 2 个工作日 | 模块负责人 -> 项目 Owner |

## 3) 执行规则

1. 任一阻塞创建后必须在 `TODO` 的 Blockers 表登记，并指定 `到期时间` 与 `严重级别`。
2. 到达 `决策时限` 仍未关闭时，必须按“逾期升级路径”升级，并在 Run Log 留痕。
3. 使用 break-glass 的阻塞必须记录票号、责任人、原因、撤销条件，且默认纳入后续治理任务。
4. 任何“先跳过检查”不能作为完成标准，只能作为有时限的临时措施。

## 5) 微小改动门槛（治理补充）

为避免“小改动也被当成全量发布文档”导致的噪音，约定以下微小改动门槛：

1. 同一轮改动仅触达单一主模块，且改动文件不超过 4 个实现文件（文档和测试文件不计入该阈值），可标记为“微小改动”。
2. 标记“微小改动”后，`delivery-doc-sync` 至少需要同步三项：交付总览、接口说明（若涉及 API）、测试说明；其余章节可在 Run Log 中注明“不适用”。
3. 标记“微小改动”后，`fullstack-test-matrix` 至少需要覆盖三类证据：正常路径、异常路径、边界路径；可通过同一测试文件完成，但必须可检索到对应场景关键词。
4. 任何微小改动仍不得直接跳过主守卫，不得将 break-glass 作为默认完成路径；若使用豁免，必须记录截止时间和撤销动作。
5. 微小改动完成后，必须在 `TODO` 的 Blockers 与 Run Log 中回填“触发门槛、验证命令、验证时间”三项证据。
6. 对“仅文档治理/守卫复验”任务，Run Log 需显式标注 `frontend: n/a` 与 `extJson: n/a`，说明本轮无页面与扩展字段改动，避免误判为漏改。
7. 文档证据文件命名尽量避免 `route/router/api/schema/model` 等实现层关键词；若任务仅为治理留痕，优先使用中性命名（如 `handoff`、`baseline`），降低路径规则误报概率。
