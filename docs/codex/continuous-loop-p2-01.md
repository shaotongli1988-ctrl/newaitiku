# P2-01 持续循环续航策略

## 背景
`P0/P1` 已完成后，连续开发不应停在“等待下一条任务”，需要进入可持续补货与节奏控制阶段，确保 Codex 能继续自主推进。

## 本轮目标
- 把“下一步”从口令驱动升级为可执行规则。
- 在不触碰高风险改动确认线的前提下，持续产出可提交的小闭环。

## 续航执行规则
1. 任务补货规则
- 当 `P0/P1` 全部 `done` 时，自动创建 `P2` 候选任务，来源限定为：守卫告警收敛、测试补强、文档治理、低风险重构。
- 每次至少保留 `2` 条 `todo` 状态的 `P2` 任务，避免下一轮无任务可选。

2. 任务选择规则
- 只选择“可在单轮内闭环”的最小任务，优先级顺序：守卫稳定性 > 测试稳定性 > 文档一致性 > 小型重构。
- 命中数据库迁移、权限模型、删除逻辑时，仍按全局约束暂停确认，不自动越权推进。

3. 证据闭环规则
- 每轮固定输出：路由 handoff、`unified_delivery_guard` 的 `batch/final` 结果、`TODO` 状态回填、原子提交。
- 若 `final` 出现告警，必须先修复再进入下一轮，不以“备注解释”替代修复。

4. 节奏与停止条件
- 默认连续推进，直到出现以下任一条件：
  - 当日达到 `10` 次提交；
  - 出现外部依赖阻塞且无法通过同轮替代任务绕过；
  - 用户明确要求切换到开发后终检阶段。

## 产出检查点
- `docs/codex/TODO.codex.md` 已新增 `P2` 任务并更新运行日志。
- 本文档作为后续“下一步”轮次的固定决策依据。
