文档定位：

- 供产品、运营、录题、研发、测试统一口径使用。

## 1. 官方政策关键信息

### 1.1 执行时间

- 优化调整后的考试科类及考试科目，自 2026 年普通专升本考试开始实行。
- 2024 年、2025 年普通专升本考试仍按原考试科类及考试科目命题、考试。

### 1.2 考试总分结构

- 总分：500 分
- 公共文化考试：2 科
- 政治：100 分
- 英语：100 分
- 专业综合（职业适应性）考核：2 科
- 专业考核科目 1：150 分
- 专业考核科目 2：150 分

### 1.3 2026 版考试大类

官方通知将原有 8 个考试大类调整为 10 个考试大类：

| 序号 | 考试大类 |
| --- | --- |
| 1 | 经济学类 |
| 2 | 管理学类 |
| 3 | 理工类 |
| 4 | 农学类 |
| 5 | 教育学类 |
| 6 | 文学类 |
| 7 | 法学类 |
| 8 | 历史学类 |
| 9 | 医学类 |
| 10 | 艺术类 |

### 1.4 题库建设必须遵守的政策口径

- 考生报考的本科专业应与专科（高职）专业大体对应。
- 联考专业按官方公布口径执行，不建议在系统内自行拆散或重命名。
- 新的考试说明是后续知识点拆分的依据，题库应预留版本化和批量导入能力。
- 本次公布的联考专业根据现有招生专业制定，当年如有新增或停招专业，以考试当年官方公布口径为准。

## 2. 题库内容对象定义

为方便后期做题库、标签、组卷、学情分析和 AI 推荐，建议把这份通知落成以下内容对象。

| 对象 | 说明 | 建议关键字段 |
| --- | --- | --- |
| 政策版本 `policy_version` | 锁定哪一版考试政策 | `version_code`、`title`、`effective_year`、`source_file` |
| 考试大类 `exam_category` | 10 个 L1 大类 | `category_code`、`category_name`、`sort_no`、`enabled` |
| 联考专业组 `joint_exam_group` | 大类下的本科联考专业集合 | `group_code`、`group_name`、`category_code`、`major_list_text` |
| 公共课科目 `public_subject` | 政治、英语 | `subject_code`、`subject_name`、`score` |
| 专业考核科目 `professional_subject` | 每个联考专业组对应 2 门专业科目 | `subject_code`、`subject_name`、`subject_slot`、`score` |
| 专科专业类映射 `vocational_mapping` | 专科专业类到联考专业组的映射 | `vocational_code`、`vocational_name`、`group_code` |
| 知识模块 `knowledge_module` | 大纲章节、模块、考点 | `module_code`、`module_name`、`parent_subject_code`、`version` |
| 题目 `question` | 题库实际题目 | `question_id`、`subject_code`、`module_code`、`question_type`、`difficulty` |

### 2.1 对象关系建议

建议按下面的关系组织：

- `policy_version` 1 对多 `exam_category`
- `exam_category` 1 对多 `joint_exam_group`
- `joint_exam_group` 1 对 2 `professional_subject`
- `public_subject` 作为全员公共科目独立维护
- `professional_subject` 1 对多 `knowledge_module`
- `knowledge_module` 1 对多 `question`
- `joint_exam_group` 多对多 `vocational_mapping`

### 2.2 为什么不能只按“大类”建题库

只按“理工类”“医学类”建题库会丢失真实考试结构。官方口径实际是：

- 全体考生统一有公共课：政治、英语
- 专业课不是只看大类，而是要落到某个“联考专业组”
- 同一个大类下，不同联考专业组的专业课科目可能完全不同
- 同一个专科专业类，也可能出现在多个联考专业组中

因此后端和前端都不应只保存 `category`，还应保存 `joint_exam_group`。

## 3. 推荐内容层级

建议题库按 4 层组织，而不是只做简单的“科目树”：

| 层级 | 名称 | 说明 | 示例 |
| --- | --- | --- | --- |
| L0 | 政策版本层 | 区分 2025 旧版与 2026 新版 | `HB_ZSB_2026` |
| L1 | 考试大类层 | 10 个考试大类 | `理工类` |
| L2 | 联考专业组层 | 大类下的专业组 | `理工3` |
| L3 | 考试科目层 | 公共课或专业课 | `信息技术概论` |
| L4 | 知识模块层 | 大纲章节、考点、能力点 | `数据结构基础` |

题目标注建议至少挂到：

- 1 个 `L0`
- 1 个 `L1`
- 1 个 `L2`
- 1 个 `L3`
- 1 个 `L4`

## 6. 附件 1 对题库和报名系统的实际影响

官方附件 1 不是简单的“专业树”，而是一张“本科联考专业组 <-> 专科专业类”的映射表。对系统设计影响很大：

- 它不是单纯的题库目录，而是报名资格和目标专业归属的判断依据。
- 同一个考试大类下，会存在多个联考专业组。
- 同一个专科专业类，可能在不同联考专业组中重复出现。
- 因此系统不应只保存“专科专业类 -> 考试大类”的单值映射，应保存“专科专业类 -> 联考专业组”的映射关系。

### 6.1 适合落地的数据结构

建议建一张独立映射表：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `policy_version` | 例如 `HB_ZSB_2026` |
| `category_code` | 考试大类编码 |
| `joint_exam_group_code` | 联考专业组编码 |
| `undergrad_major_name` | 本科联考专业名称 |
| `vocational_major_code` | 专科专业类编码 |
| `vocational_major_name` | 专科专业类名称 |
| `source_notice` | 来源文号 |
| `enabled` | 是否启用 |

### 6.2 业务上会直接用到的场景

- 学生首次注册时，根据专科专业或目标本科专业，匹配默认联考专业组。
- 报考大类变更时，不仅要变更大类，还要重新绑定联考专业组和专业课科目。
- 题库筛选、模考组卷、错题本统计、学情分析，都应基于联考专业组而不是只基于大类。

## 7. 建议的题库字段口径

每道题至少建议保留以下字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| `policyVersion` | `HB_ZSB_2026` | 对应哪版考试政策 |
| `examCategoryCode` | `SCIENCE_ENGINEERING` | 考试大类 |
| `jointExamGroupCode` | `SCIENCE_3` | 联考专业组 |
| `subjectCode` | `INFO_TECH_INTRO` | 具体考试科目 |
| `subjectType` | `PUBLIC` / `PROFESSIONAL_1` / `PROFESSIONAL_2` | 科目类型 |
| `moduleCode` | `INFO_001` | 知识模块 |
| `questionType` | `single_choice` | 题型 |
| `difficulty` | `0.65` | 难度系数 |
| `sourceType` | `official` / `mock` / `ai_generated` | 题目来源 |
| `applicableGroups` | `["SCIENCE_3"]` | 适用专业组 |

## 8. 对现有系统的直接落地建议

这份通知适合直接作为以下模块的基准：

- 后台“科目大类配置”
- 后台“题库科目树”
- 学生注册后的“报考大类/联考专业组绑定”
- 模拟考试组卷规则
- 错题本按大类、专业组、科目、知识点筛选
- AI 推荐题和学情分析的统计维度

建议优先顺序：

1. 先落 `考试大类 + 联考专业组 + 科目` 三层字典。
2. 再根据后续考试说明补齐 `知识模块`。
3. 最后导入题目，避免题目录入后再大规模迁移标签。
